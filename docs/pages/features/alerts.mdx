When a delivery attempt fails to deliver an event to a destination, an alert can be triggered. Alerts are scoped per destination.

Alerts notify the operator of failures and can optionally auto-disable destinations using `ALERT_AUTO_DISABLE_DESTINATION`.

It's your responsibility to format and deliver the alert to your tenant using your existing notification system.

## Configuration

| Config | Description | Default |
| --- | --- | --- |
| `ALERT_CONSECUTIVE_FAILURE_COUNT` | Number of consecutive failures before the 100% threshold | `100` |
| `ALERT_AUTO_DISABLE_DESTINATION` | Auto-disable destinations at 100% failure threshold | `false` |
| `ALERT_EXHAUSTED_RETRIES_WINDOW_SECONDS` | Deduplication window for exhausted retry alerts (seconds) | `3600` |

## How It Works

Alert evaluation runs within the log batch processor with per-entry ack/nack for at-least-once delivery.

### Consecutive failure alerts

The `ALERT_CONSECUTIVE_FAILURE_COUNT` setting defines the threshold. Alerts are emitted when the consecutive failure count reaches **50%, 70%, 90%, and 100%** of this threshold. A successful delivery resets the counter.

At the 100% threshold, the destination is auto-disabled if `ALERT_AUTO_DISABLE_DESTINATION=true`.

### Exhausted retry alerts

When a delivery exhausts all retry attempts, an `alert.attempt.exhausted_retries` event is emitted. These are deduplicated per event+destination within a configurable time window.

## Receiving Alerts

Alerts are delivered through the [Operation Events](./operation-events) system. Configure a sink (HTTP, AWS SQS, GCP Pub/Sub, or RabbitMQ) and subscribe to alert topics:

```
OPERATION_EVENTS_TOPICS=alert.destination.consecutive_failure,alert.destination.disabled,alert.attempt.exhausted_retries
OPERATION_EVENTS_HTTP_URL=https://example.com/alerts
OPERATION_EVENTS_HTTP_SIGNING_SECRET=your-secret
```

See [Operation Events](./operation-events) for full configuration and payload details.
